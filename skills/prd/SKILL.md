# PRD — Product Requirements Document

> Status: scaffold. The full workflow for this skill will be authored next.

## Purpose

Help the user produce a clear, complete Product Requirements Document — focused on
the *what* and *why* of a product or feature from the user's and business's
perspective, not the engineering implementation.

## When to use

- The user is starting a new feature or product and needs to define requirements.
- The user has a rough idea and wants it shaped into a structured PRD.
- The user explicitly asks for a PRD.

## Process (to be expanded)

1. Clarify the problem and the user/customer it affects.
2. Define the proposed solution from the user's perspective.
3. Capture user stories and acceptance criteria.
4. Define success metrics.
5. Mark what is explicitly out of scope.

## Template (to be expanded)

- Problem statement
- Target users
- Goals & success metrics
- Solution overview
- User stories / requirements
- Out of scope
- Open questions
