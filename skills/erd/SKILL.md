# ERD — Engineering Requirements Document

> Status: scaffold. The full workflow for this skill will be authored next.

## Purpose

Help the user produce a clear, complete Engineering Requirements Document — focused
on the *how*: the technical approach, architecture, data, and trade-offs needed to
build what a product spec describes.

## When to use

- The user has a PRD (or a defined feature) and needs to plan the engineering work.
- The user wants the technical design captured before implementation.
- The user explicitly asks for an ERD or technical design doc.

## Process (to be expanded)

1. Establish the requirements and constraints being solved for.
2. Define the proposed architecture and key components.
3. Specify data models, interfaces, and API contracts.
4. Identify dependencies, risks, and trade-offs.
5. Outline testing, rollout, and observability plans.

## Template (to be expanded)

- Context & requirements
- Proposed architecture
- Data model
- Interfaces / API contracts
- Dependencies & risks
- Testing strategy
- Rollout plan
- Open questions
